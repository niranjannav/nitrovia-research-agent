## Your task
Analyse the workbook structure and produce a SchemaMapping JSON object with these exact fields:

```json
{
  "domain": "<domain string>",
  "mapping_name": "<short human-readable name, e.g. 'SMILK Sales Weekly v1'>",
  "confidence": <0.0–1.0 float, your overall confidence in this mapping>,
  "notes": "<any important observations about the file format>",

  "source_sheets": ["<sheet name 1>", "<sheet name 2>"],
  // List ONLY transactional/raw-data sheets. Exclude summary/pivot/lookup sheets.
  // For multi-year files, include sheets for all years.

  "header_row": <1-based integer, which row contains column headers>,

  "column_roles": {
    // Map standard field names to the EXACT column header string in the Excel.
    // Only include fields you are confident about. Omit fields not present.
    // Standard fields depend on domain — see field_definitions above.
    "<standard_field>": "<exact Excel column header>"
  },

  "time_structure": {
    "type": "<one of: wide_monthly | wide_weekly | long_date_col | quarterly_pivot | annual_only | dual_metric_wide_monthly | transposed_financial>",
    // wide_monthly: separate columns for each month (Jan–Dec), one row per record
    // wide_weekly: separate columns for each week number, one row per record
    // long_date_col: single date column, one row per transaction
    // quarterly_pivot: separate Q1/Q2/Q3/Q4 columns
    // annual_only: single annual total column
    // dual_metric_wide_monthly: multi-row header; row 0 has sparse metric labels
    //   (e.g. "Qty" spanning cols 6-17, "Lts" spanning cols 19-30), row 1 has
    //   duplicate month names. The sheet notes will say "dual_metric_header: ...".
    // transposed_financial: rows = financial line items, columns = monthly dates.
    //   The sheet notes will say "transposed_financial: ...".

    "columns": ["<col header 1>", "<col header 2>"],
    // For wide_monthly/wide_weekly/dual_metric_wide_monthly: list ALL time period column headers in order.
    // For long_date_col: ["<date column header>"]
    // For quarterly_pivot: ["Q1", "Q2", "Q3", "Q4"] (or actual header names)
    // For transposed_financial: omit or leave empty.

    "year_source": "<one of: sheet_name_suffix | explicit_column | single_year | unknown>",
    // sheet_name_suffix: year is in the sheet name (e.g. "Data-Qty-2024" → 2024)
    // explicit_column: there is a year column in the data
    // single_year: the entire file covers one year (state it in year_value)
    // unknown: cannot determine

    "year_value": <integer or null>,
    // Only set if year_source is single_year. Otherwise null.

    // ── dual_metric_wide_monthly additional fields ──
    "metric_blocks": [
      // REQUIRED for dual_metric_wide_monthly. List each metric block:
      {
        "label": "<metric label from row 0, e.g. 'Qty'>",
        "start_col": <0-based column index where this block starts>,
        "end_col": <0-based column index where this block ends (inclusive)>,
        "maps_to": "<standard field name, e.g. 'quantity_units' | 'quantity_litres' | 'revenue'>"
      }
    ],

    // ── transposed_financial additional fields ──
    "label_column": "<column letter containing row labels, e.g. 'B'>",
    // REQUIRED for transposed_financial.

    "row_label_map": {
      // REQUIRED for transposed_financial. Map standard field names to the EXACT
      // row label string in the label_column as it appears in the Excel sheet.
      // Example:
      "<standard_field_name>": "<exact Excel row label>"
    },
    // Standard finance field names: revenue, cogs, gross_profit, gross_profit_pct,
    // other_income, operating_expenses, operating_profit, ebit, ebitda, net_income,
    // litres_sold, revenue_per_litre, cost_per_litre,
    // total_assets, total_liabilities, total_equity,
    // operating_cash_flow, investing_cash_flow, financing_cash_flow, net_cash_flow

    "date_header_row": <1-based row number containing date/year column headers>
    // REQUIRED for transposed_financial. Check the sheet notes for "date_header_rows".
  },

  "derived_fields": {
    // Optional. Fields that must be computed from other columns.
    // Omit this section if no derived fields are needed.
    "<derived_field_name>": {
      "from": "<source column header>",
      "method": "<one of: prefix_2chars | prefix_Nchars | regex | lookup_table | companion_sheet_join | constant>",

      "pattern": "<regex string, only for regex method>",
      "map": {"<source_value>": "<derived_value>"},  // for prefix or regex methods
      "lookup_sheet": "<sheet name>",                // for lookup_table method
      "lookup_key_col": "<column header>",           // for lookup_table method
      "lookup_value_col": "<column header>",         // for lookup_table method
      "companion_sheet_pattern": "<pattern>",        // e.g. "Data2-Lts-{year}" for companion_sheet_join

  },

  "exclude_sheets": ["<sheet name>"],
  // Sheets to skip entirely (summaries, pivots, instructions, lookup tables).

  "warnings": ["<warning string>"]
  // Any data quality issues, ambiguities, or assumptions you made.
}
```

## Rules
1. Only include `source_sheets` that contain row-level transactional data. Pivot/summary sheets (those with SUMIFS, totals by region/channel/product) should go into `exclude_sheets`.
2. Use the exact column header strings as they appear in the workbook (case-sensitive).
3. If the time structure uses wide_monthly columns named like "01 - January", list all of them in `time_structure.columns` in order.
4. If the year is embedded in the sheet name (e.g. "Data-Qty-2024"), set `year_source` to `sheet_name_suffix`.
5. For derived fields: if the customer/product code prefix encodes a category (e.g. "PO305" → POS channel), describe the prefix mapping completely.
6. Set `confidence` honestly. If the structure is ambiguous, say so in `warnings`.
7. Return ONLY valid JSON. No prose, no markdown fences, no explanation outside the JSON object.
